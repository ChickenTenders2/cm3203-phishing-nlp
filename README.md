# CM3203 Phishing Email Detection

This project compares NLP approaches to phishing email classification, from simple TF-IDF pipelines to fine-tuned transformers. It is written as part of the CM3203 final-year dissertation module at Cardiff University.

The central question driving it is whether more complex models actually justify their cost. When a TF-IDF + LinearSVM trained in seconds can hit 99% accuracy on one dataset and 98.5% on another, you have to ask what you gain from something 66 million parameters large. The answer turns out to be modest on raw accuracy but more meaningful on false positive rate, which matters considerably more in a real email filtering context than a leaderboard position.

## Datasets

Two datasets are used, and they are quite different from each other.

**Kaggle Human-LLM Generated Phishing/Legitimate Emails** combines real phishing emails written by humans with synthetic ones generated by large language models, across both phishing and legitimate classes. After deduplication and preprocessing, 2,392 emails remain (1,724 legitimate, 668 phishing). It is a small dataset by NLP standards — 1,435 training examples and 957 in the test set — and the class imbalance means F1 on the phishing class is a more informative figure than overall accuracy. One thing that makes it unusual is that it lets you separately evaluate how well models generalise to LLM-written emails versus human-written ones, since the source is tracked throughout.

**MeAJOR** is a much larger multi-source corpus assembled from several public email collections. After deduplication and cleaning, 104,572 emails remain (57,947 legitimate, 46,625 phishing), split into 62,743 training and 41,829 test examples. The class balance is tighter here and the variety in sender domains and writing styles makes it both harder and more representative of what a deployed filter would encounter.

Both datasets use stratified 60/40 train/test splits with `random_state=42` throughout.

## Preprocessing

For the Kaggle data, emails from the human-generated source have their subject and body fields joined into a single text input. LLM-generated rows already come with a pre-built `text` column, so those are left untouched. After merging the four source files, 1,202 duplicate texts are removed and 240 rows with no recoverable content are dropped.

For MeAJOR, subject and body are joined with explicit `Subject:` and `Body:` prefixes before deduplication. Roughly 4,112 duplicate text-label pairs are removed from the original 108,685-row raw dataset. A single row with a missing label is also discarded.

All processed splits are saved as Parquet files under `data/processed/` and are not tracked in git given their size.

## Models

Seven model families are trained and evaluated across both datasets.

**TF-IDF + classical classifiers.** Four classifiers are trained on top of TF-IDF features using a 50,000-term vocabulary with unigrams and bigrams and sublinear TF scaling applied. The vectoriser is always fitted on training data only to avoid leakage. The four classifiers are Multinomial Naive Bayes, Logistic Regression, Linear SVM, and Random Forest (100 trees).

**SimHash + kNN.** A locality-sensitive hashing baseline that compresses TF-IDF vectors into binary fingerprints using random projections, then classifies by k-nearest neighbour using Hamming distance between fingerprints. On the Kaggle set, 64-bit and 128-bit fingerprints and k values from 1 to 9 are all evaluated on a held-out validation slice before picking the best configuration (128-bit, k=1) for final test evaluation.

**TextCNN.** Parallel convolutional filters of sizes 2, 3, and 4 run over learned token embeddings, each followed by global max pooling, then concatenated before a dropout and classification layer. Good at picking up fixed short phrases that appear often in phishing emails. Word embeddings are randomly initialised and trained end-to-end.

**BiLSTM.** A two-layer bidirectional LSTM that reads each token sequence in both directions and concatenates the final hidden states before classification. The MeAJOR version includes an additional dense hidden layer before the output, since the larger training set can support a slightly deeper architecture. The core recurrent component is the same in both.

**DistilBERT.** The `distilbert-base-uncased` checkpoint is fine-tuned for 5 epochs using AdamW with a learning rate of 2e-5, weight decay of 0.01, and a 10% linear warmup over total training steps. Sequences are truncated to 256 tokens. The pre-trained tokeniser is used without modification — no vocabulary fitting on the email data.

The Kaggle deep learning experiments use PyTorch; the MeAJOR deep learning experiments use Keras/TensorFlow. This was an intentional choice to confirm that the evaluation framework works the same regardless of backend, since all metrics are computed through scikit-learn after predictions are collected. All inference timing excludes DataLoader overhead and is averaged across three runs.

## Results

Metrics are reported for accuracy, precision, recall, F1 on the phishing class, false positive rate (legitimate emails misclassified as phishing), and inference time per email. Consolidated result tables are generated in `results/tables/results_summary.ipynb` and exported as CSV files in `results/tables/`.

A few things are worth pulling out. On the Kaggle dataset, LinearSVM comes very close to matching DistilBERT on F1 (0.983 vs 0.979) while being roughly four orders of magnitude faster at inference. The per-source breakdown tells a more revealing story: MultinomialNB effectively fails on LLM-generated phishing emails, scoring an F1 of 0.06 on that sub-group compared to 0.80 on human-written ones. LinearSVM and RandomForest both score perfect on LLM-generated test emails, which suggests those models are finding real structural differences in the phishing content rather than just mimicking surface patterns from the training distribution.

On MeAJOR the picture tightens up considerably. LinearSVM at 0.983 F1 is only about six points behind DistilBERT at 0.989, but DistilBERT's false positive rate of 0.007 is roughly half of LinearSVM's 0.015. Across 41,829 test emails that translates to around 290 incorrectly flagged legitimate emails for DistilBERT versus around 625 for LinearSVM — the kind of difference that would matter a great deal to users receiving false spam verdicts on real mail.

SimHash underperforms considerably on MeAJOR compared to its Kaggle results. The best configuration (128-bit, k=9) reaches 0.844 F1 with an FPR of 0.120 — roughly ten times worse than any of the classical TF-IDF models. The gap is likely explained by dataset scale: with 62,743 training emails, the kNN search space is large enough that Hamming distance on 128-bit fingerprints loses enough resolution to cause frequent misclassifications near decision boundaries. The hyperparameter sweep also shows a clear trend where increasing k keeps pushing the FPR down at the cost of recall, which suggests the fingerprints are not discriminative enough on this dataset for a small neighbourhood to be reliable.

## Repository structure

```
data/
  raw/               original source files (not tracked)
  processed/         cleaned Parquet splits (not tracked)
notebooks/
  kaggle_preparation.ipynb
  kaggle_classical_baselines.ipynb
  kaggle_deep_learning.ipynb
  kaggle_distilbert.ipynb
  kaggle_simhash.ipynb
  meajor_preparation.ipynb
  meajor_classical_baselines.ipynb
  meajor_deep_learning.ipynb
  meajor_distilbert.ipynb
  meajor_simhash.ipynb
results/
  figures/           confusion matrix and training curve images
  metrics/           per-model CSV result files
  tables/            consolidated summary notebook and exported comparison tables
models/
  classical/         saved model artifacts
```

The `results/` directory is now organised by output type:

- `results/figures/` stores confusion matrices, threshold sweeps, and training curves.
- `results/metrics/` stores raw per-notebook CSV outputs, including `all_results.csv`.
- `results/tables/` stores the summary notebook (`results_summary.ipynb`) plus generated comparison tables such as `kaggle_summary_table.csv`, `meajor_summary_table.csv`, and `all_results_normalised.csv`.

## Setup

```bash
pip install -r requirements.txt
pip install transformers tensorflow seaborn
```

The `requirements.txt` covers the core dependencies used across the notebooks, including pandas, Parquet support, PyTorch, scikit-learn, matplotlib, and `transformers`. `tensorflow` is still required separately for the MeAJOR deep learning notebook. `seaborn` is optional and only used in a few visualisation cells.

Notebooks should be run in order within each dataset group: the preparation notebook first, then classical baselines, deep learning, DistilBERT, and SimHash. The preparation notebook must be run before any of the modelling notebooks since it produces the Parquet splits everything else loads from.
